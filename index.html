查询计划&优化 Part1

本篇文章服务于CMU15-445 lec 13查询优化第一节，主要讲述基于启发式（规则式）的优化方法

Overview

为什么有查询优化呢？

因为SQL语句是声明式的，查询语句只告诉DBMS去执行什么，而不管其怎么去执行；不同的查询计划之间的效率是有很大的不同的，因此DBMS需要找到一种高效的查询计划。

大致上查询优化从上层分为两种：

  1. 启发式的，使用一种静态的规则，遇到什么就去怎么做，DBMS并不知道使用了这种规则后查询计划的性能时候更好，它只是按照我们约定的规则去操作的。
  2. 基于代价式的，我们需要评估每个查询计划的代价，然后根据代价去选择更小的查询计划。

逻辑计划 VS 物理计划

逻辑计划是从上层进行描述的，比如说进行join

物理计划是从实操的层面上得，比如是next loop join 还是 hash join

优化器会从逻辑上的代数表达式生成一个实际的物理表达式；这两个表达式大致是相当的

每一个物理的算子都有一个特定的执行策略，其依赖于数据的物理格式，物理和逻辑上得计划并不是一 一对应的。

SQL执行的过程

  1. 当应用程序发来了SQL语句时，先经过SQL重写，在文本字符串上做一些优化
  2. 将重写的SQL语句发给解析器，解析器会生成抽象的语法树
  3. 语法树经过Binder，将索引与实际物理相结合，生成逻辑执行计划
  4. 然后经过树重写，将系统表的信息录入后再生成进一步的逻辑计划，最后送进优化器
  5. 优化器经过代价评估最后生成物理的执行计划。

Relational Algebra Equivalences

当两个算数表达式产生结果，我们称其为等价的

DBMS无需经过代价模型，可以直接求得较好的算数表达式，我们有时候称其为查询重写

Predicate pushdown

Relational Algebra Equivalences

  • X = Y AND Y = 3 ⇒ X = 3 AND Y =3
  • 投影算子尽量尽早的执行，将复杂的算子往下移，能最大成都减少复杂性
  • Join算子支持交换律和结合律，不同的顺序对开销影响很大

Projection Pushdown

Logical Query Optimization

把一个逻辑的查询计划转化为等价的逻辑查询计划，增加列举最优计划的可能性

  1. Split Conjunction Predicates

    解剖连接谓词，把连接在一块的谓词分开

    比如将一连串的多个选择转化分别转化为几个选择，并且如果选择算子选择的属性只在某个表中出现，可以将这个选择放到这个表的

  2. Replace Cartesian Products with Joins

    我们可以将笛卡尔积换成内部的连接

  3. Projection Pushdown

    如果说投影算子所投影的属性只在某个表中出现，则可以讲这个投影算子下移到这个表中

Nested Queries

把可以提前计算好的先计算好，然后替换掉

  1. 重新来去除他们之间的相关性，或者是展开他们
  2. 把嵌套的子查询分开然后把结果存储在暂时的表中

Expression Rewriting

优化器会将查询表达式转化为最优的表达式

比如 1 = 0，这种简单的直接去掉

查询计划&优化 II

本篇文章服务于CMU15-445 lec 14查询优化的第二节，主要讲述基于代价式的优化方法

Cost Estimations

DBMS会用代价评估模型去估算执行一个计划的代价，这些计划是等价的（能够输出相同的结果），然后去选择代价最小的计划作为最终的执行路径

代价评估主要依托于：

  • CPU：代价很小，但是很难去评估
  • Disk IO：共需要传输多少磁盘块
  • Memory：使用了多少DRAM
  • Network：有多少讯息需要传递

代价评估不可能详尽的列举所有查询计划的方案，因为这样太慢了，比如说Join算法，其具有结合律和交换律，加入说有4个表需要join，就有$4^n$个连接方案，优化器不可能将所有的组合走一遍的。

为了估计查询的代价，DBMS维护了一些内部的信息，这些内部信息是关于表，属性和索引的。为什么要维护这些内部信息呢？为了避免重复的计算，不同的系统内部存储的信息是不同的，我们不能拿不同DBMS的维护的信息去做比较，这些信息会在后台进行更新。

对于每一个表R，我们有以下统计信息

  • $N_R$ : R中tuple的数量
  • $V (A,R)$ : 在表中R中，属性为A的值得数量

由上面的信息，我们可以得到 选择基数 SC(selection cardinality) 统计量，这个选择基数是属性为A的记录的平均数量，记作 $N_R/V(A,R)$，我们这个公式是基于数据的一致性的， 但是事实情况并不是如此的。

Selection Statistics

选择基数可以用于确定特性输入的数量

如果谓词是相等的谓词估计，这将很简单，但是如果是范围选择， 这样会复杂一些

谓词P的选择是符号谓词条件的元组所在的分数，对于复杂谓词的选择很难去评估其准确性，并且其也有可能会产生某些问题，

我们发现，谓词的选择与符合该谓词的元组的概率是相当的。因此很多概率得的公式会被用于谓词的计算上，这在处理复杂谓词的的选择率上是很有用的。假如说我们假设进行并的谓词是独立的，那么概率的公式可以被用于谓词的计算，我们计算总的选择性可以用单个的选择性去相乘。

Selectivity Computation Assumptions

在计算谓词的选择概率上是，我们需要遵守以下的规则

  • Uniform data：均匀分布，我们假设值得分布式均匀的。
  • Independent Prediactes：谓词的属性都是独立的
  • Inclusion Principle：连接键是重叠的，内部表的属性在外部表也都能找到。

Histograms

真实的数据通常是有偏差的，很难做出假设，但是数据集是重要的，我们不能舍弃，我们为了节省内存，我们可以采用直方图的方式去把数据分组

我们可以设置等宽的直方图去存储数据

我们可以用等深的直方图去存储数据，可以改变桶的宽度，使得每个桶中出现的次数大致相同的。

Sampling

DBMS可以使用采样将用来估计谓词的选择性，这样会使得样本量变小，如果基础表的更改量超过了某个阈值，那么DBMS就会自动更新样本

Plan Enumeration

基于代价式的查询计划的分类

Single-Relation Query Plans

对于单表查询，最大的障碍是如何设置存取方法， 比如是是顺序扫描，索引扫描，或者二分查找等等。大多数数据库都是使用的启发式的，而不是基于代价式的模型

对于OLTP事务型的查询，这是非常简单的，因为他们是基于索引搜索的，其必定存在最佳的索引搜索，这可以是简单的启发式的。

Multi-Relation Query Plans

对于多表查询，随着表的数量增加，可以选择的Join组合增长的很快的，所以我们需要设定固定的搜索空间，IBM System R涉及处了左深度搜索，这是因为左深度收搜更适合管道模型， 因为DBMS不需要具体的连接运算符输出，如果DBMS的优化器仅仅考虑左深度搜索，这样会减少很多的查询时间。

  1. 枚举所有的关系表排序，
  2. 枚举连接算法的选择，比如HJ或者NLJ
  3. 枚举访问方法找到代价最小的路径。

查询执行 Part 2

本篇文章服务于CMU15-445 lec12，主要讲述查询执行的过程中如何处理并发机制

Background

我们假定之前讨论的查询都是基于单线程的，但是在实际的应用过程中，查询执行都是并发执行的

并发执行的优点：

  • 吞吐量增加（单位时间内处理的SQL数量）
  • latency（处理每条SQL语句需要更少的时间）
  • 从客户端来看，响应速度变快，可用性提高
  • TCO降低（total cost of ownership）总成本降低了

并发执行分为另种，查询间并发执行（inter-query parallelism）和 查询内并发执行（intra-parallelism）

Parallel vs Distributed Databases

我们区别一下并发和分布式的概念

相同点：并发和分布式都是将数据库分布在多个资源中以提高其并发性，比如CPU core，CPU socket , CPUs，disks，memory，无论是并发还是分布式，其结果与在当个的DBMS上执行的结果相同。

不同点：

  • 并发的DBMS中，资源在物理上是非常是非常紧密的，资源之间的通讯速度很快，交互的成本也非常低并且可靠
  • 分布式DBMS中，资源之间物理上离的非常远，资源之间的通讯是要通过网络的，较慢，资源之间的交互成本是需要着重考虑的。

Process Model

处理模型：系统如何支撑来着多用户应用程序、环境的并发请求
works：DBMS就是由一个个work组成的，这些work处理来着客户端的请求，并且返回结果
当应用程序发来请求时，这些请求就会被分配给不同的工人

Process Per Worker

这里的process都是操作系统的进程，每一个worker都会被分配一个OS的进程

  • 依赖于OS的调度
  • 共享内存会用到全局的数据结构中，进程之间通过贡献内存来进行通信
  • 其中一个进程的crash不会似的整个系统崩溃，但是同时也是非常吃资源的。
  • IBM DB2， pg, Oracle

当请求来的时候，经过分配器分给每个work，每个work是一个进程，然后处理数据，处理完数据后再返回请求

Process Pool

worker可以使用进程池中任意空闲的进程

  • 依赖OS的调度，并且进程之间共享内存
  • 对于CPU的局部缓冲不利，
  • IBM DB2 Pg(2015)

进程之间共享内存，任务来的时候，由分配器去 进程池中选择空闲的进程进程分配，因此在查询的时候相同的查询可能不是同样的进程，因此对CPU的局部性利用率不高

Thread Per Worker

每个数据库仅仅有一个进程，这个进程有很多线程，此时DBMS可以自己管理线程，而不用操作系统的介入了，一个线程一个worker的可以使用也可以不适用分配器

  • DBMS可以自己调度线程
  • 线程的crash可能会使得整个系统崩溃
  • 上下文切换的时候开销更小，没有必要维护共享模型，因此资源本来就是用进程进行调度的，并不一定支持查询内的并行执行
  • IBM DB2 MSSQL MySQL Orac

Scheduling

对于每一个查询计划，DBMS都必须去决when，where，and how 去执行

  • 有多少task？
  • 多少CPU cores 可以用
  • 任务在哪个CPU核上可以使用
  • 输出应该输出在哪里

DBMS知道的比OS多很多

Execution Parallelism

Inter-Query Parallelism（查询间并发执行）

DBMS并发的执行不同的查询，因为多个worker同时运行，执行不同的任务，因此整体的性能提高了，增大了吞吐量，减少了延迟

如何查询是只读的，那将会有很好的并发性，但是如果涉及到修改，则会有更加复杂的冲突

Intra-Query Parallelism（查询内并发执行）

单个的查询并发的执行，这样会减少一条语句的执行时间，减少延迟性

  1. Inter-Operator（Horizontal）

    将算子分成相互独立的片段，然后分别在不同且不想交的数据集上执行相同的操作，最后使用exchange算子将其合并或者分解。

    通常来说exchange算子包括一下几个

    • gather：将多个worker的结果合并成同一个输出流，在DBMS最常用的类型
    • repartition：由多个输入，也有多个输出，以一种方式输入，然后以另外一种方式输出去重新分配他们
    • distribute：讲一个简单的输入流分成多个输出流

    查询计划被分成不相交的阶段，给定的计划由不同的worker操作，在操作片段上同时调用next，然后从他们各自的页面上检索数据

  2. Inter-Operator(Vertical)

    流水线执行，操作符之间重叠，以便无需物化就可以将操作数从一个阶段传递到下个阶段

  3. Bushy

    第三种方法是将上面两种方法合并，即有水平方向的，又有垂直方向的，

I/O Parallelism

在数据库的执行过程中，磁盘才是瓶颈，如何保证磁盘的并发性呢？

Multi-Disk Parallelism

RAID0：将数据分在不同的磁盘上，如下图所示，可以并发的取页面1,2,3,；此时是有OS决定的，对DBMS来说是透明的，其并不知道底层发生了什么

这种一旦其中一个页面损害，则会导致真个文件无用，还有一种方法是RAIO1：是将文件复制到不同的磁盘上，即存副本，存副本会导致资源的利用率过低

Database Partitioning

讲数据库分解成不同的物理段，物理段存在不同的位置，在逻辑上是完成的，应用程序可以在逻辑上访问这些数据，但是并不知道其在内部的结构。

如何分区呢？

  1. 垂直分区

    将表的属性存储在单独的位置，为了重建原始记录，我们还需要记录一些元组信息

  2. 水平分区

    将标的元组根据分区键被分成不同的相交的段，如根据散列，范围和位置分区。
