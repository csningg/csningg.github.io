vsag框架概要

VSAG论文解读

论文介绍了一个名为 VSAG (Vector Search Algorithm Graph) 的开源框架，由蚂蚁集团（Ant Group）开发 。它的核心目标是解决当前主流的基于图的近似最近邻搜索（ANNS）算法（如 HNSW）在生产环境中面临的性能瓶颈。简单来说，VSAG 是对现有图算法（Graph-based ANNS）的一套全方位工业级优化方案。

尽管 HNSW 等基于图的算法在召回率和性能上表现优异，但在大规模生产环境中存在三大痛点 ：

  1. 随机内存访问导致的高延迟： 图遍历需要在内存中随机跳转，导致 CPU 的 L3 缓存未命中率（Cache Miss Rate）极高，严重拖慢速度。
  2. 距离计算开销大： 高维向量的距离计算非常消耗 CPU 资源。
  3. 调参成本过高： 图索引对参数（如最大度数、候选集大小）非常敏感。传统调参需要反复重建索引，耗时极长（可能需要数天），导致很多系统只能使用次优参数。

针对以上三个问题，论文依次提出了解决方案，以下是对该论文的解读，包括其主要思想、整体流程以及巧妙设计的解读。

VSAG 主要思想

1. 三大设计优化

VSAG 针对上述问题提出了三个维度的优化：

内存访问优化 (Memory Access Optimization)

这是 VSAG 最底层的创新，旨在让“随机”的图搜索变得对 CPU 缓存更友好。

  • 确定性访问贪婪搜索 (Deterministic Access Greedy Search)： 传统的贪婪搜索是串行的（计算一个->取下一个）。VSAG 将其改为批量处理，对邻居节点进行分组，并利用软件预取（Software Prefetching）技术，在计算当前节点距离时，提前将下一个节点的数据加载到缓存中。
  • 部分冗余存储 (PRS - Partial Redundant Storage)： VSAG 设计了一种特殊的存储结构。它不仅存储节点本身，还将该节点的压缩后的邻居向量直接冗余存储在节点旁边。
    • 作用： 这样一来，访问某个节点的邻居时，内存访问就从“随机跳转”变成了“顺序读取”。这激活了 CPU 的 硬件预取（Hardware Prefetching） 机制，极大降低了缓存未命中率。
    • 平衡： 引入了参数 $\delta$（冗余比率）来平衡内存占用和 CPU 效率。$\delta=1$时全冗余（速度最快，费内存），$\delta=0$ 时无冗余（省内存）

自动参数调优 (Automatic Parameter Tuning)

VSAG 不需要用户手动痛苦地调参，也不需要为了调参反复重建索引。

  • 基于“掩码/标签”的索引压缩技术： VSAG 先用较宽松的参数构建一个“大索引”，并在每条边上打上标签（Label），标记这条边在不同参数下是否应该存在。
  • 免重建调优： 在调优或搜索时，只需根据标签过滤边，就能模拟出不同参数构建的索引效果。这使得索引级参数（Index-level Parameters） 的调优时间从“数天”缩短为“几乎即时” 。
  • 查询级自适应： 使用决策树模型判断查询的难易程度，动态调整搜索时的候选集大小（$ef_s$）。简单的查询少搜点，复杂的查询多搜点。

距离计算加速 (Distance Computation Optimization)

  • 标量量化 (Scalar Quantization, SQ) + SIMD： 将浮点向量压缩（如 SQ4/SQ8），利用 AVX-512 指令集并行计算，大幅减少计算量。
  • 距离分解 (Distance Decomposition)： 预先计算向量的模长（Norm）并缓存，将欧氏距离计算简化为内积运算，减少实时计算指令。
  • 选择性重排序 (Selective Re-rank)： 先用低精度（量化后）向量快速筛选出一批候选者，再对其中极少部分进行高精度重算，兼顾了速度和准确率。

2. 整体流程

Figure 1 展示了 VSAG（Vector Search Algorithm Graph） 的整个搜索框架流程。它将整个搜索过程分为了三个主要阶段：存储（PRS）、搜索执行（Deterministic Access Greedy Search） 和 结果优化（Selective Re-rank），同时贯穿了自动参数调优。

以下是对这张图的分步详细解释：

第一步：左侧模块 —— PRS (Partial Redundant Storage, 部分冗余存储)

这是 VSAG 的数据存储基础，旨在优化内存访问效率。

  • 混合存储结构 (Redundant vs. Flat): 图中展示了两种存储状态。
    • Original Vector (蓝色): 存储原始的高精度向量数据。
    • Redundant Neighbors (绿色): 这是核心设计。VSAG 将某个节点的邻居向量（压缩后的）直接冗余存储在该节点旁边。
    • $\delta = 0.5$ (Redundancy Ratio): 这是一个调节参数。图中 $\delta=0.5$表示只有 50% 的数据采用了冗余存储（Redundant Storage），剩下的 50% 采用普通存储（Flat Storage)。这是一种在“内存占用”和“计算速度”之间取得平衡的机制。
  • Code Layout (数据布局): 放大镜展示了邻居节点具体存了什么：
    • Quantized Code: 量化后的低精度向量代码（例如 4-bit 量化），用于快速计算。
    • Id: 邻居的 ID。
    • Norm: 预先计算好的向量模长，用于加速距离计算（距离分解技术）。
    • Metric Transform: 距离变换逻辑。

第二步：中间模块 —— Deterministic Access Greedy Search (确定性访问贪婪搜索)

这是搜索执行的核心阶段，主要解决图搜索中“内存随机访问”导致的 Cache Miss 问题。

  • Async Prefetch During Computing (计算时异步预取): 顶部的框图展示了流水线工作模式。当 CPU 正在计算当前向量的量化代码（Q Code Computing）时，利用空闲带宽同时预取（Prefetching） 下一批需要的数据到缓存中。
  • Only Prefetch Valid Neighbors (只预取有效邻居): 中间的 $x_1$到 $x_{10}$ 序列展示了过滤逻辑。算法会先剔除掉那些已经访问过的或者被剪枝的邻居（图中 $x_2, x_3$被跳过），只对有效的邻居进行预取和计算。
  • Stride Prefetch (步长预取): 底部的箭头（例如从 $x_1$指向 $x_6$）展示了步长预取策略。为了防止数据还没加载进缓存 CPU 就要用，VSAG 会“看得很远”。例如，在计算 $x_1$ 的距离时，它发出的指令是预取 $x_6$ 的数据（步长为 5），从而掩盖内存延迟。

第三步：右侧模块 —— Selective Re-rank (选择性重排序)

这一步是为了弥补第二步中使用“低精度量化向量”带来的精度损失。

  • Result Push Heap (结果入堆): 第二步计算出的邻居节点会被推入一个结果堆中。
  • Low vs. High Precision:
    • 在搜索过程中使用低精度（Quantized Code）快速筛选。
    • 只有通过筛选的少量优质候选项，才会回读原始向量（Original Vector）进行高精度距离计算（即 Re-rank），确保最终结果准确。
  • Pop Heap and Visit: 底部蓝色的长箭头表示循环。从堆中弹出当前最近的节点，回到左侧 PRS 读取它的邻居，开始下一轮搜索 。

第四步：最右侧 —— Self-Tuned Parameters (自动参数调优)

在整个框架运行过程中，有一个自动调优器在持续工作。

  • 三层参数 (Environment/Query/Index): 图例展示了 VSAG 针对三个层级的参数进行自动化管理
    • Index-level: 如图的最大度数，通过“掩码”技术在不重建索引的情况下动态调整。
    • Query-level: 如搜索时的候选集大小 ($ef_s$)，根据查询难度动态调整。
    • Environment-level: 如预取步长 ($\omega$)，根据硬件环境自动调整。

VSAG 不是一种全新的图算法结构，而是一个针对图算法的“生产级加速器”。

它保留了图算法（如 HNSW）的高召回率特性，但通过魔改内存布局（PRS）解决了“缓存未命中”这一最大的性能杀手，并通过标签化索引解决了“调参难”的运维痛点。对于需要大规模部署向量数据库的企业来说，VSAG 是比原生 HNSWlib 更高效、更可控的选择。

预取技术

下图完整描述了 VSAG 如何通过特定的内存布局（PRS）配合流水线式的搜索策略（Stride Prefetch）来“喂饱”CPU，最后通过重排序（Re-rank）保证精度，并利用自动调优来适应不同场景。

这张图（Figure 2）直观地展示了 VSAG 如何利用 软件预取（Software-based Prefetch)技术，将传统的“串行等待”内存访问模式优化为“并行流水线”模式，从而显著减少 CPU 的空转等待时间。

以下是详细的分步解释：

图例说明

  • 橙色方块 (Fetch - Sync Operation): 代表同步的内存读取。在基准算法中，当 CPU 需要某个向量数据但缓存中没有时（Cache Miss），必须停下来等待数据从内存加载完毕，这会阻塞后续操作 。
  • 蓝色方块 ($\tau(x, x_q)$ - Sync Operation): 代表同步的计算操作。这里指计算向量 $x$与查询向量 $x_q$ 之间的距离（如欧氏距离)。
  • 绿色方块 (Prefetch - Async Operation): 代表异步的预取指令。这是 VSAG 引入的关键优化。它是一条告诉 CPU “提前把数据从内存拉到缓存”的指令，它在后台运行，不会阻塞当前的计算任务。

1. 上半部分：Baselines (传统基准方法的痛点)

这是现有普通图搜索算法（如未优化的 HNSW）的执行流程：

  • 模式： Fetch $\rightarrow$ Compute $\rightarrow$ Fetch $\rightarrow$ Compute。
  • 过程：
    1. 先花时间取 $x_1$ 的数据（橙色 Fetch）。
    2. 数据到了才能算 $x_1$的距离（蓝色 $\tau$）。
    3. 算完后，再去花时间取 $x_2$ 的数据（橙色 Fetch）。
    4. 数据到了再算 $x_2$ 的距离…
  • 缺陷： 这种串行模式导致 CPU 频繁陷入等待。每当处理一个新邻居节点，CPU 都要停下来等内存数据（即图中橙色方块占用的时间），导致了大量的时间浪费。

2. 下半部分：Ours (VSAG 的优化方案)

这是 VSAG 采用软件预取后的执行流程，核心在于将“取数据”的时间重叠在“算数据”的时间里：

  • $t_1$ (初始加载): 处理第一个节点 $x_1$ 时，不可避免地需要先从内存获取数据（Fetch）。
  • $t_2$ (并行流水线 - 关键点):
    • CPU 开始计算 $x_1$ 的距离（蓝色块）。
    • 与此同时，VSAG 发出了一条异步指令 Prefetch x2（绿色块），让内存控制器在后台去加载下一个节点 $x_2$ 的数据 。
    • 因为预取是异步的，它没有打断 $x_1$ 的计算，两者并行发生。
  • $t_3$ (无缝衔接):
    • 当 CPU 准备处理 $x_2$ 时，由于在 $t_2$ 阶段已经提前发出了预取请求，$x_2$ 的数据此时已经加载到了 L3 缓存中 。
    • 因此，CPU 不需要像 Baseline 那样执行橙色的 Fetch 等待，而是直接开始计算 x_2 的距离。
    • 同时，利用计算 $x_2$ 的这段时间，再次预取下一个节点 $x_3$。
  • $t_4$: 同样，$x_3$ 的数据也已就绪，直接进行计算。

结论：Saving 2 timeslots (节省了 2 个时间片)

通过这种“吃着碗里的（计算当前），看着锅里的（预取下一个）”的策略，VSAG 成功消除了 $x_2$ 和 $x_3$ 的内存读取等待时间。

  • 效果： 如右下角所示，原本 Baseline 需要 6 个时间片（$t_1$ 到 $t_6$）才能完成的任务，VSAG 仅用了 4 个时间片（$t_1$ 到 $t_4$）就完成了，节省了 $t_5$ 和 $t_6$ 。
  • 这解释了为什么 VSAG 能大幅降低 L3 Cache Miss Rate 并提升 QPS。

参数自动寻优

1. 查询级参数调优 (QLP)：集成了 GBDT 决策树模型

查询级参数（Query-level Parameters, QLP）主要指的是搜索时的**候选集大小 $ef_s$**（在伪代码中是 $L$）。这个参数对查询速度和召回率的影响最大。

  • 问题： 不同的查询向量（Query Vector）搜索难度差异巨大。对简单的查询使用很大的 $ef_s$ 会浪费时间；对困难的查询使用很小的 $ef_s$ 会牺牲召回率。
  • VSAG 的解决方案： 训练一个轻量级的分类器，实时判断查询的难度，动态调整 $ef_s$。

模型细节：

  1. 使用的模型： GBDT (Gradient Boosting Decision Tree，梯度提升决策树)。 论文选择 GBDT 是因为它计算速度快、模型小巧，适合在低延迟的搜索路径中实时推理。
  2. 训练目标： 模型的目标是预测一个查询是“简单”还是“困难”，或者更细致地分为 $k$ 个难度等级。
  3. 特征工程： 模型使用在搜索初期阶段（前几跳）就能获取到的局部特征作为输入，例如：
    • 当前候选集的距离分布： 候选者之间的距离方差、最大最小值。
    • 距离变化率： 上一轮和本轮最远距离的变化。
    • 节点连通度： 当前探索的节点的度数等。
  4. 动态调整：
    • 当模型预测当前查询属于“简单”类别时，系统会动态地将 $ef_s$ 设置为一个较小的值 ($L_{min}$)，提前结束不必要的探索。
    • 当模型预测为“困难”类别时，系统会增大 $ef_s$ ($L_{max}$)，允许算法探索更多的邻居，以确保高召回率。

2. 索引级参数调优 (ILP)：基于“标签”的免重建技术（非模型）

索引级参数（Index-level Parameters, ILP）如最大度数 $M$、建图时的候选集大小 $ef_{construction}$ 等。调整这些参数传统上需要重建整个图索引。

  • 问题： 重建索引非常耗时（如 GIST1M 需要数十小时）。
  • VSAG 的解决方案： 基于“标签”的虚拟索引技术，实现参数调优的“免重建”。

技术细节：

  1. 一次性构建： VSAG 只用一套最宽松的参数（例如最大的 $M$）构建一次索引。
  2. 边标签（Edge Labeling）： 在构建过程中，VSAG 不会物理删除那些被剪枝的边。相反，它会给每条边打上一个标签（Label），记录这条边在多严格的剪枝条件下会被保留。
  3. 动态过滤： 在搜索阶段，用户可以动态设置一个虚拟参数。系统会根据这个虚拟参数，实时地在图遍历时过滤掉那些标签不满足要求的边。
  4. 结论： 这是一种工程优化而非机器学习模型，它将原本需要大量时间进行 I/O 操作（重建索引）的调优过程，转化为了极快的内存内数据过滤操作。

3. 环境级参数调优 (ELP)：基于网格搜索（非模型）

环境级参数（Environment-level Parameters, ELP）主要与底层硬件和运行环境相关，例如 **软件预取步长 $\omega$**。

  • 问题： 不同的 CPU 架构、缓存大小、内存带宽，其最优的预取步长 $\omega$ 是不同的。
  • VSAG 的解决方案： 离线网格搜索（Grid Search）。

技术细节：

  1. 测试： 在索引构建完毕后，VSAG 在一个预定义的参数网格中，运行少量的测试查询，系统性地尝试 $\omega$ 的不同取值。
  2. 评估： 评估哪个 $\omega$ 值能带来最高的 QPS（即最低的 L3 Cache Miss Rate）。
  3. 锁定： 一旦找到最优的 $\omega$ 值，就将其锁定为当前运行环境的配置参数。

这同样是一种系统配置自动化的方法，不依赖复杂的机器学习模型，但能确保底层硬件的效率得到最大化。

VSAG伪码解读

VSAG 论文中的算法 1：确定性访问贪婪搜索（Deterministic Access Greedy Search）

从表面上看，这段代码与标准的基于图的近似最近邻搜索（如 HNSW）中的贪婪搜索逻辑非常相似。然而，VSAG 的全部性能提升和设计优势，都隐藏在算法中关键操作的底层实现细节中。

以下是对该算法的详细解释：

1. 算法概述与标准流程

该算法描述了在 VSAG 构建的图索引 ($G$) 上，从一个入口点开始搜索，最终找到最接近查询向量 ($q$) 的 L 个候选者（即 $ef_s$）的过程。

  • 初始化（Lines 1-3）： 从预先确定的入口点 $e$ 开始搜索。将 $e$ 加入到两个优先级队列：
    • $C$ (Candidate Pool)：最终候选集，大小限制为 L。
    • $W$ (Working Queue)：待探索节点集合，用于驱动搜索。
  • 主循环（Lines 4-14）： 只要 $W$ 中还有未探索的节点，就继续循环。
  • 探索（Line 5）： 从 $W$ 中取出当前离查询向量 $q$ 最近的节点 $x_c$。

2. VSAG 核心优化实现（三层加速）

VSAG 的三大优化（内存、计算、调参）都被嵌入在主循环内部（Lines 6-14）的操作中：

优化 A：内存访问加速 (Deterministic Access & PRS)

VSAG 针对图搜索的随机内存访问模式进行了改造，尽管伪代码是线性循环，但底层实现是并行的。

  • 邻居获取（Line 6）： 获取 $x_c$ 的所有邻居 $N(x_c)$。
    • PRS 机制： 在实际实现中，由于 PRS（部分冗余存储） 的内存布局，当程序访问 $x_c$ 的数据时，其部分邻居的压缩向量（$Q_{x_i}$）已经被 硬件预取器 自动加载到 Cache 中了。
  • 批量处理与预取（Lines 7-14 的实际实现）：
    • 在执行循环前，VSAG 会先将所有有效邻居（未访问且未被剪枝）收集成一个批次。
    • 在计算 $x_j$ 的距离时，底层实现会利用 软件预取（Stride Prefetch） 指令，同时异步加载下一个批次中 $x_{j+\omega}$ 的数据。这实现了计算与内存 I/O 的重叠（如图 Figure 2 所示）。

优化 B：计算加速 (Low-Precision Distance)

距离计算是高维搜索中的 CPU 瓶颈。VSAG 在 Line 10 的距离计算 $d_{c,j}$ 中应用了加速技术：

  • 低精度计算： 这里的距离计算使用的不是原始的高精度（Float32）向量，而是 PRS 中存储的低精度量化代码（Quantized Code）。这极大减少了计算量。
  • 距离分解与 SIMD： 实际的距离函数还使用了距离分解（预先计算 $||y||^2$）来简化欧氏距离的计算，并利用 SIMD 指令（如 AVX-512）进行高度并行计算，进一步提高 $d_{c,j}$ 的计算速度。

优化 C：自动参数调优 (Self-Tuned Parameters)

算法中的两个关键参数和判断条件都受到了 VSAG 自动调优机制的动态影响：

  • 动态 L（Line 13）：
    • L 是最终候选集 C 的大小限制（对应于 $ef_s$ 参数）。在传统算法中这是固定的。
    • 在 VSAG 中，这个参数由 查询级参数调优 (QLP) 机制控制。QLP 使用一个 GBDT 决策树模型，根据查询 $q$ 的难度动态调整 $L$ 的值。简单的查询会设置较小的 $L$，困难的查询会设置较大的 L，以平衡召回率和速度。
  • 动态剪枝条件（Line 8）：
    • $\text{IsPruned}(x_j)$ 判断邻居 $x_j$ 是否应该被剪枝。
    • 这个判断结合了 索引级参数调优 (ILP) 机制。VSAG 在建图时给每条边打了标签，**$\text{IsPruned}$ 函数在运行时根据标签动态过滤边**，模拟不同的索引结构，避免了耗时的索引重建。

总结

尽管算法 1 的骨架是经典的贪婪搜索，但 VSAG 赋予了其中关键操作全新的、高效的实现：

Line 8 检查是否剪枝 ILP (索引标签过滤) 动态调优索引参数，无需重建

VSAG 相比现有优秀算法（如 HNSWlib）在哪里？

相比于目前工业界标准的 HNSWlib 和 Facebook 的 Faiss，VSAG 的优势主要体现在以下几个方面：

显著的性能提升 (高达 4 倍 QPS)

在相同的召回率（Recall）下，VSAG 的吞吐量（QPS）显著高于 HNSWlib。

  • 数据： 在 GIST1M 数据集上，同等精度下 VSAG 比 HNSWlib 快 4倍 。
  • 原因： HNSWlib 的 L3 Cache Miss 率通常高达 90% 以上，而 VSAG 通过 PRS 和预取技术将其降低到了 39% 左右。

“免重建”的参数自适应

  • 存量痛点： 在 HNSWlib 中，如果你想调整图的连通度（M）来优化性能，你必须重新插入所有数据重建索引。
  • VSAG 优势： VSAG 可以在不重建索引的情况下，通过“边过滤”动态调整索引结构参数。对于拥有十亿级向量的生产系统，这节省了数小时甚至数天的运维时间。

更好的资源利用率平衡

  • 存量痛点： 传统算法通常受限于机器规格（例如 4核16G），经常出现 CPU 跑满但内存空闲，或者内存满了 CPU 却在空转。
  • VSAG 优势： 通过 PRS 的冗余比率 $\delta$，VSAG 可以根据机器的实际硬件（计算型 vs 内存型实例）灵活调整。如果是计算型机器，增加冗余以换取速度；如果是内存受限机器，减少冗余以节省空间。

对高维数据的处理更优

  • VSAG 优势： 针对 OpenAI 等大模型生成的 1536 维 embedding，VSAG 的量化和距离优化表现尤为突出。实验显示在 OpenAI 数据集上，VSAG 提供了比 HNSWlib 高约 400% 的 QPS 16。
