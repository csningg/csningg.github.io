Task #1 LRU replacement policy

需要实现以下四个函数：

  • Victim：将某个页面从缓冲池中清除
  • pin：给某个页面加钉，使其不能够被Victim
  • unpin：给某个页面解钉，使其可以被Victim
  • Size：返回此时缓冲池中的页面数量

准备工作

需要先在头文件中，声明成员变量（因为不好想应该设计哪些变量才能完成上述函数，我觉得这部分是较难的）

  1. 阅读 replace.h 文件，抽象类 Replacer 告诉了我们每个函数应该做什么，返回什么值
  2. 阅读 LRUReplacer.h 文件，一定要看好注释，此时你应该知道有个 max_nums_pages _应该被定义
  3. 思考LRU算法的实现，可参考网上教程，此处我们采用 list 和 unordered_map 来实现LRU算法
  4. 因为需要保证线程安全，需要加锁，此处我们使用 lock_guard，不懂请点击

开始设计

list 和 unordered_map 的参数类型是什么呢？

list 维护的是缓冲池中frame的id；unordered_map 是为了实现O(1) 时间复杂度的 list 中地址的查找， 因此 key 应该和 list 类型一直， value 应该为 key在链表中的地址

构造函数：

直接存储变量的值即可

Victim：

  • 明白参数是什么？参数是用来存储移除页面的id的，不是移除这个页面！！！
  • 此函数仅需要实现 lru 算法中的删除部分，不用考虑其他的，最后把自己绕进去了
  • 注意边界条件(文档说了)，清除哪个页面呢？自然是链表的最后一个元素，因为它不经常用，清除完注意 lru_map_的同步
  • 细节：list的back返回的是引用，等号右边是引用，左边应该是list的类型的变量。list 中的erase 的参数要求 是list 的迭代器，list的类型不行，而map 中的erase参数可以是键值对，也可以是key的类型

Pin：

  • 此处pin某个页面，是将该页面删除，unpin某个页面是将某页面添加至链表和哈希表中，emmm 尝试着去理解吧
  • 首先我们应该保证此页面在缓冲池中，而后通过 frame_id 将 lru_list_ 和 lru_map_ 中对应的元素删除，注意erase的参数，list 要求是迭代器，map可以是key，也可以是键值对

Unpin:

  • 就是把frame_id 插入的 lru_list_ 和 lru_map_中去
  • 边界条件：是否已经存在或者已经满了（满的话可以考虑移除一个元素，也可以直接返回，此处我们为求简单，直接返回）
  • list 的插入头结点 直接用 push_front，unorderer_map 直接赋值插入，记得第二个参数是迭代器！

Size：

  • 返回此时缓冲池的大小

Task 2 buffer pool manager instance

🦾此处再次强调读文档，很重要，函数前面的注释和函数原型的注释，非常重要，非常重要！

本次task是实现缓冲池管理器，需要实现六个函数，每个函数具体做什么，我们后续细说

  • FlushPgImp ：刷新某个页面到磁盘中
  • UnpinPgImp ：将某个页面解锁
  • FetchPgImp ：从空闲表中获取某个页面
  • NewPgImp ：从缓冲池中获取某个页面
  • DeletePgImg ：删除某个页面
  • FlushAllPagesImp ：刷新所有的页面到磁盘中

在开始之前，有几个概念我们需要强调一下，

  1. 每个page对象包含一块内存，DiskManager将使用它作为一个位置来复制从磁盘上读取的物理页面的内容，在page的整个生命周期中，其可能包含不同的物理页面。其用数组pages_来保存，用什么来定位呢？用frame_id 来定位找到不同的page
  2. page_table_存放的是page_id与frame_id 的映射，每一个page都有自己的page_id,而每个page又必须用frame_id 去找到;可以这样理解，假如有个数组，frame_id 是数组的下标，而page_id 是数组元素的地址。注意此时是类比，并不真的是地址
  3. free_list 存放的是空闲的page，其用frame_id 去定位
  4. 在Replacer中的pin，是将页面从lru的数据结构提出，Unpin是又将其加上，

**bool FlushPgImp(page_id_t page_id)**：

  • 我们需要将 id 是page_id 的页面刷新到磁盘中去。注意参数是干什么的？参数是用来定位page的。如果成功就返回true，失败就返回false
  • 如何找到这个page呢？所有的page都必须通过pages_数组找到，因此必须先找到frame_id
  • 边界条件，如果是page_id是无效，或者buffer pool中根本没有这个页面
  • 对于WritePage 可以去对应文件中查看其参数，注意是脏页的标记更改！

void FlushAllPgsImp() ：

  • 上面的 FlushPgImp 会写，这个一定会写，如果不会，说明上面还没搞明白，继续理解上面的函数
  • 这边有一点我在做的时候有疑问😒，如果没有可以直接跳过，我起初对这行代码不理解 Page* page = &pages_[frame_id]; Page *pages_`表明 pages_是一个数组，强调强调是数组
  • 自己的疑问，如果同时把* 和 & 去掉，从理论上应该是可以理解的，但是实际却编译不通过，为什么？

Page NewPgImp(page_id_t *page_id) ：

  • 此函数让我们从内存中找到一个page，然后给它脱胎换骨，然后将它返回，参数是用来存放脱胎换骨够的页面的id的
  • 页面的数量是一定的，要么在free_list中，要么可以调用Replacer中的Victim找到一个，如果都没有，那说明没有页面可以被清楚了
  • 最后调用 allocatePage 重新分配一个page id，page本就由这一系列成员函数和成员变量组成的。
  • 然后更新元数据，在page中的元数据，自己查看哪些需要修改。
  • 更新page_table,将此页面其加入到page_table，那么其pin_count 是不是要增加，是不是需要pin

Page *FetchPgImp(page_id_t page_id) :

  • 此函数让我们从缓冲池中取某个页面，参数是用来定位page的，如果定位不到，那么它就是我们新page的id
  • 如果前面的函数你都懂了，看着函数提示，你一定能做出来。
  • page_table 和 pin 和 pin_count 都是同步变化的
  • 填充page的内容，此处涉及读磁盘，然后将其写到page中，请细细品！

bool DeletePgImp(page_id_t page_id)：

  • 从缓冲池中删除一个页面，参数仍然是为了定位page的
  • 仍然按照提示去写，你一定可以的！

bool UnpinPgImp(page_id_t page_id, bool is_dirty) ：

  • 从缓冲池里Unpin一个页面， is_dirty 是用来修改 page的参数的。

Task 3 PARALLEL BUFFER POOL MANAGER

先上结论，第三个实验总体来说比较简单，有几个提醒的点，其余的都很好做

  • 需要定义成员变量，成员变量请看构造函数

  • 有很多缓冲池管理器，可以采用数组的方式来进行管理，数组类型为 BufferPoolManagerInstance* 这样是为了后面我们能调用成员函数，如何定位呢，用取模的方式，文档中有说

  • ParallelBufferPoolManager 和 BufferPoolManagerInstance都是继承自 BufferPoolManager 二者是同级的关系

  • 注意构造函数需要new 数组，那么析构函数就需要将这些数组delete掉
